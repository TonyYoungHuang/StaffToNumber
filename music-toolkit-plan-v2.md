# 无线谱互转工具（SEO 官网 + 中国大陆交付 Web）产品与实施方案 v2.0

> 本版本用于替代此前“三端一体（Web + 桌面 + 小程序）”的大方案，收敛为一个更容易落地、验证和商业化的版本。

---

## 0. 版本结论（先看这个）

### 0.1 本次方向确认
- **不做微信小程序**
- **当前阶段只做 Web，不做桌面客户端**
- **产品核心只做“五线谱与简谱之间的互转”**
- **一期确认只做两条能力链路：**
  1. `MusicXML -> 简谱`
  2. `简谱文本输入 -> 五线谱`
- **移调功能不进入一期**
- **移调功能进入下一阶段，作为同一个网站里的独立模块实现**

### 0.2 产品形态确认
本项目拆为两个面向不同场景的 Web 入口，但共用同一套核心转换后端：

1. **SEO 官网（面向 Google 搜索）**
   - 用于品牌展示、能力说明、内容获客、搜索流量承接
   - 目标是让用户能够通过 Google 搜索找到网站
   - 主要承担“发现你、理解你、进入工具”的作用

2. **交付应用站（面向中国大陆用户购买后使用）**
   - 用于登录、下单、上传、转换、下载、查看历史
   - 以 PC 端使用为核心，页面按电脑操作习惯设计
   - 这是实际承载付费和工具使用的主应用

> 简单理解：**一个官网负责被搜到，一个应用站负责真正使用。**

## 1. 项目定位

### 1.1 项目名称建议
- 中文：**无线谱互转工具**
- 英文对外名可选：**Score Convertor / Staff & Numbered Notation Converter**

### 1.2 项目定位
这是一个面向 PC 用户的乐谱转换网站，专注解决以下核心问题：

- 将标准五线谱文件转换为简谱
- 将结构化简谱文本转换为五线谱

### 1.3 本期不再追求的目标
以下内容在 v2.0 中全部排除，不进入 MVP：

- 小程序入口
- 桌面客户端
- PDF/图片/扫描件自动识别
- OCR 乐谱识别
- 移调
- 曲库
- 会员体系
- 多端同步
- 复杂在线编辑器

> 核心原则：**先把“互转”做成可用产品，再扩功能。**

## 2. 用户与场景

### 2.1 目标用户
- 音乐老师
- 编曲/制谱从业者
- 乐器培训机构
- 合唱/乐队排练组织者
- 有简谱/五线谱互转需求的个人学习者

### 2.2 典型使用场景

#### 场景 A：五线谱转简谱
用户手上已有标准 `MusicXML` 五线谱文件，希望快速生成练习用简谱版本。

#### 场景 B：简谱转五线谱
用户手上有简谱内容，但不是图片，而是可以规范输入的简谱文本，希望生成标准五线谱文件或可打印版本。

#### 场景 C：官网获客 -> 进入应用
用户通过 Google 搜索“staff to numbered notation”、“jianpu converter”、“MusicXML to numbered notation”等关键词找到官网，了解产品后进入工具站使用。

#### 场景 D：中国大陆用户购买后使用
用户在中国大陆完成购买或开通后，通过交付应用站在电脑端上传文件、提交转换任务并下载结果。

## 4. 核心产品决策

### 4.1 为什么不做小程序
- 手机屏幕不适合查看和校对乐谱
- 上传、预览、打印、下载的完整流程更适合电脑端
- 小程序会显著增加适配成本，但不能增强核心体验

### 4.2 为什么只做 PC 优先 Web
- 开发成本低于多端
- 更新部署方便
- 中国大陆用户也更容易使用电脑完成专业操作
- 后续如果需要桌面程序，可在 Web 成熟后再封装为桌面壳

### 4.3 为什么移调放到下一期
- 当前核心问题还不是“功能不够多”，而是“主功能尚未形成闭环”
- 五线谱/简谱互转与移调虽然相关，但应分阶段做
- 先做互转，能更快建立产品价值和用户反馈

### 4.4 移调功能的产品归属
移调功能建议：

- **不独立做一个新网站**
- **不在一期并行开发**
- **在下一期作为同网站中的独立模块实现**

建议未来网站结构：
- 五线谱转简谱
- 简谱转五线谱
- 五线谱移调
- 格式转换（后续可选）

> 结论：**移调保留在同一网站体系内，但作为二期独立模块。**

## 5. 输入与输出范围定义

这是本项目成败的关键，需要在第一版就写清楚。

### 5.1 五线谱转简谱：一期支持范围

#### 输入
- 仅支持 `MusicXML`

#### 输出
- 网页预览版简谱
- 可下载简谱文件（建议优先支持 PDF）
- 后续可扩展导出为图片或结构化文本

#### 说明
- 一期默认只做**旋律型简谱**
- 对复杂复调、和声密集、钢琴双手谱等场景不承诺完全自动高质量转换

### 5.2 简谱转五线谱：一期支持范围

#### 输入
- 仅支持**结构化简谱文本输入**

#### 推荐输入结构
- 调号
- 拍号
- 速度（可选）
- 小节线
- 音高数字
- 时值标记
- 升降号/附点/连音等基础符号

#### 输出
- 网页预览版五线谱
- 可下载五线谱 PDF
- 可下载 `MusicXML`

### 5.3 一期明确不支持的输入
- PDF 乐谱
- 扫描图片
- 手机拍照图片
- 非结构化简谱截图

> 必须在官网和工具页清楚写出：**第一版只支持标准文件和结构化输入，不支持 OCR 识别。**

## 6. 产品结构设计

### 6.1 整体结构

```text
SEO 官网（Google 收录）
   └─ 引导进入
交付应用站（PC Web）
   ├─ 登录/购买后使用
   ├─ 上传文件
   ├─ 创建转换任务
   ├─ 查看结果
   └─ 下载文件

统一后端 API
   ├─ files
   ├─ jobs
   ├─ auth（可后置）
   └─ billing（后续）

转换 Worker
   ├─ MusicXML -> 简谱
   └─ 简谱文本 -> 五线谱
```

### 6.2 推荐部署形态
- `www.xxx.com`：SEO 官网
- `app.xxx.com`：交付应用站
- 共用同一套 API 与转换服务

这样做的好处：
- 官网更利于搜索引擎抓取
- 工具站更利于做登录和业务流程
- 营销内容和操作界面分离，体验更清晰

## 7. 技术方案建议

### 7.1 推荐技术栈

#### 前端
- 官网：`Next.js`（适合 SEO、静态化、内容页）
- 应用站：`Next.js` 或同技术栈下独立应用

#### 后端
- `Node.js + NestJS` 或同类服务框架

#### 任务队列
- `Redis + BullMQ`

#### 数据库
- 开发阶段：`SQLite` 或 `PostgreSQL`
- 生产阶段：建议 `PostgreSQL`

#### 文件存储
- 开发阶段：本地存储
- 生产阶段：对象存储或稳定文件存储服务

### 7.2 核心工程思想
- 官网与应用站分离
- 转换统一走 `jobs`
- 前端不直接做重计算
- 所有结果文件统一管理和下载
- 为二期移调模块预留任务扩展能力

## 8. 转换引擎设计

### 8.1 统一内部乐谱模型
建议在系统内部抽象一个统一的 `ScoreDocument` 模型，作为两种转换方向之间的中间层。

#### 作用
- `MusicXML` 先解析为内部结构
- 简谱文本也先解析为内部结构
- 再由内部结构渲染为目标格式

#### 好处
- 两条转换链路共用同一套基础数据结构
- 二期加入移调时可以直接复用
- 后续增加更多输出格式也更容易

### 8.2 一期转换链路

#### 链路 A：MusicXML -> 简谱
`MusicXML Parser -> ScoreDocument -> Jianpu Renderer -> Preview/PDF`

#### 链路 B：简谱文本 -> 五线谱
`Jianpu Text Parser -> ScoreDocument -> Staff Renderer -> MusicXML/PDF`

### 8.3 二期移调链路预留
`MusicXML Parser -> ScoreDocument -> Transpose Module -> Renderer`

> 这也是为什么移调不应该单独再做一个站点，因为底层解析与渲染体系高度复用。

## 12. 中国大陆购买与使用方案

### 12.1 推荐方案
中国大陆用户在一期建议使用**PC Web 应用站**，而不是单独开发桌面程序。

### 12.2 原因
- 与当前“只做 Web”的方向一致
- 开发速度快
- 更新和修复更方便
- 电脑端已能满足专业使用场景
- 后续如果确实需要离线能力，再封装桌面壳

### 12.3 中国大陆交付站要求
- 中文界面优先
- PC 端操作优先
- 支持购买后登录使用
- 稳定上传、稳定下载、稳定任务状态展示
- 支持订单状态与历史记录

### 12.4 商业化建议
一期可以采用最简模式：
- 购买后开通使用权限
- 不急于上复杂会员系统
- 优先验证用户是否愿意为“互转能力”付费

### 12.5 合规与支付
涉及域名、部署、支付接入、面向中国大陆稳定访问等事项时，需在正式上线前单独核验相关合规与渠道要求；这些内容不作为本方案的功能设计前提，但应在上线前列入上线清单。

## 14. 风险点与提前规避

### 14.1 最大技术风险
不是前端页面，而是“互转规则是否足够稳定”。

### 14.2 风险 1：简谱输入标准不统一
规避方式：
- 一期只定义一种结构化简谱输入规范
- 提供示例和模板
- 不接受自由格式文本

### 14.3 风险 2：五线谱转简谱质量不稳定
规避方式：
- 一期只承诺 `MusicXML`
- 一期只承诺旋律型简谱
- 对复杂谱例明确标出限制

### 14.4 风险 3：范围膨胀
规避方式：
- 一期只保留两个转换方向
- 移调明确后置
- OCR、PDF、桌面端全部后置
